UE RPC 完整数据流：三种方向

详解 UE RPC 三种方向的完整调用链：Server RPC (客户端->服务器)、Client RPC (服务器->拥有者客户端)、Multicast RPC (服务器->所有相关客户端)，含 Bunch 打包与 UDP 发送机制。

RPC 调用入口

UFUNCTION(Server, Reliable, WithValidation) 声明后，UHT 生成 ServerXXX_Implementation（真正逻辑）和 thunk（Validate + 走网络）。开发者调 ServerXXX() 实际进入 thunk。

RPC Bunch 结构

一个 RPC Bunch 里塞：ContentBlockHeader（目标 SubObject GUID）+ FieldHeader（RPC 在类中的 FieldNetIndex）+ Payload（参数序列化后的 bit 流）。组织成 [BlockHeader | FieldHeader | Payload]。

Server RPC（客户端 -> 服务端）

只能从拥有该 Actor 的客户端调用（AutonomousProxy 端），SimulatedProxy 端调用会被丢弃。典型场景：玩家按下开火键 -> 客户端调 ServerFire() -> 服务器权威判定。

ServerXXX_Validate - 安全校验

带 WithValidation 的 Server RPC 必须提供 ServerXXX_Validate 函数。返回 false -> 服务器直接踢出该客户端（防作弊）。

典型防作弊场景：客户端上报"我扣了敌人 999999 点血"-> Validate 检查伤害数值上限 -> false -> 踢。

服务端执行

ProcessEvent -> Implementation 做权威判定：射线检测、伤害计算、扣弹药。改了 Replicated 属性会通过下一次同步分发给所有相关客户端（含 AutonomousProxy 自己）。

Client RPC（服务端 -> 拥有者客户端）

只发给该 Actor 的 OwningConnection（AutonomousProxy 那个连接）。SimulatedProxy 端永远收不到别人的 Client RPC——数据私密性的天然保障。

典型场景：服务器判定"你被击中了"-> 调 ClientPlayHitSound() -> 只在受害玩家自己屏幕上播放特效/UI。

Multicast RPC（服务端 -> 所有相关客户端）

也遵循网络相关性——远处的客户端如果对该 Actor 无 Channel，就收不到。服务端发起时自己也执行一次 Implementation（NM_ListenServer 场景）。

延迟发送优化

源码实证：不可靠 Multicast 会先入队 QueuedExportBunches，等下一次 Actor 的 ReplicateActor 时和属性同步一起发——省 Bunch header + 增加 UDP 合批率。

源码：Runtime/Engine/Private/NetDriver.cpp:2820

注意事项

  • 服务端会为每个相关 Connection 各生成一份 Bunch——不是"一份广播"
  • 若 Multicast 需要让所有玩家状态一致：推荐做法是 服务端改 Replicated 属性 + Multicast 只做特效
  • 可靠 Bunch 存在 OutRec 链表里，直到收到 ACK 才释放；丢包会自动重发
  • 不可靠 Server RPC 不入 OutRec，丢了就丢了
