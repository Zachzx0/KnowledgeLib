UE 网络时序 · RPC 三种方向

通过时序详解 UE RPC 三种方向的全流程：Server RPC (AutonomousProxy -> Authority)、Client RPC (Authority -> AutonomousProxy)、Multicast RPC (Authority -> 所有客户端)。

三种方向对比

方向发送者接收者典型场景
Server RPCAutonomousProxyAuthority玩家开火/拾取/交互
Client RPCAuthorityAutonomousProxy伤害提示/UI效果
MulticastAuthority所有相关客户端爆炸/环境音效/广播消息

Server RPC 客户端 -> 服务端

调用入口

玩家按下开火键 -> 客户端调 ServerFire()。只能从 AutonomousProxy 端调用（SimulatedProxy 上调会被丢弃）。

UHT Thunk 检测

UHT 生成的 ServerFire 会先检查 GetNetMode() != NM_Client：服务端调 -> 直接跳到 Implementation（本地执行）；客户端调 -> 走网络路径。

RPC 上传

FUNC_NetReliable 则 Bunch.bReliable=1，存入 OutRec 链表等 ACK，丢包自动重发。不可靠 Server RPC 不入 OutRec。

Validate 安全校验

带 WithValidation 的 Server RPC 必须先跑 ServerXXX_Validate。返回 false -> 服务器直接踢出该客户端。

典型防作弊：客户端上报 "伤害 999999" -> Validate 检查上限 -> false -> 踢。

Implementation 执行

ProcessEvent -> Implementation 做权威判定：射线检测、伤害计算、扣弹药。改了 Replicated 属性会通过下一次同步分发给所有相关客户端。

Client RPC 服务端 -> 拥有者客户端

派发

通过 Actor->GetNetConnection() 找目标（通常是 PlayerController 的 NetConnection）。找不到 -> 静默丢弃。

私密性保障

只发一份到 AutonomousProxy 那个连接。SimulatedProxy 端永远收不到别人的 Client RPC——数据私密性的天然保障。

客户端执行

客户端播放伤害数字飘字、震屏、红屏效果——纯表现层，不改 Replicated 状态。

Multicast RPC 服务端 -> 所有相关客户端

队列优化

源码关键优化：不可靠 Multicast 不立即发，而是入队 QueuedExportBunches，等下一次 ReplicateActor 时和属性同步一起发——省 Bunch header + 增加 UDP 合批率。

注意事项

  • 也遵循网络相关性——远处客户端如果对该 Actor 无 Channel 就收不到
  • 服务端会为每个相关 Connection 各生成一份 Bunch（不是"一份广播"）
  • 服务端自己也执行一次 Implementation（NM_ListenServer 场景）
  • 若需要所有玩家状态一致：推荐 服务端改 Replicated 属性 + Multicast 只做特效

通用：Bunch 打包与 UDP 发送

所有 RPC 最终都走：

  1. UActorChannel::SendBunch -> 塞入 UNetConnection SendBuffer
  2. FlushNet -> 打包 PacketHeader + AckBits + Bunch 群
  3. UDP SendTo
  4. 客户端 TickDispatch -> 解包 -> 派发到对应 UActorChannel
  5. ProcessBunch -> FRepLayout::ReceivePropertiesForRPC -> ProcessEvent -> Implementation
UE 的 UDP 包永远是复合包——一个 Packet 里可能塞属性同步 Bunch + 多个 RPC Bunch。
