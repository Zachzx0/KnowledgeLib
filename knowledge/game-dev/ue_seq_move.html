UE 网络时序 · 移动预测与服务器修正

UE 网络架构最经典的三方交互——同时涉及 ServerMove RPC + ClientAdjustPosition RPC + 属性同步 + 客户端预测 + SavedMoves 回放 + SimulatedProxy 平滑插值。

核心思想：延迟不可消除，但可以被隐藏。

这套架构叫 Client-side Prediction + Server Reconciliation——从 Quake III 沿用至今，UE 是这套架构在游戏引擎中最完备的实现之一。

差错检测

ServerCheckClientError 对比 ClientLoc vs ServerLoc：

  • 差距 < MaxPositionErrorSquared 阈值 -> 认可客户端位置，不发修正
  • 差距 > 阈值 -> 判定客户端错了（碰撞不同 / 作弊 / 丢包），准备修正

分支 A：预测正确（99% 常见路径）

如果 ClientLoc ~= ServerLoc，服务端不发修正 RPC，只把新位置作为 Replicated 属性推给其他客户端（SimulatedProxy）。AutonomousProxy 端本地预测的位置就是最终位置——玩家感觉零延迟。

分支 B：位置误差大 -> 服务器修正

例：客户端认为自己冲进了门，但服务端判定门被锁上了 -> 服务端要"拉回"客户端。

ClientAdjustPosition (Reliable RPC)

ACharacter::ClientAdjustPosition_Implementation -> GetCharacterMovement()->ClientAdjustPosition_Implementation(TimeStamp, NewLoc, NewVel, ...)。Reliable 保证一定到达。

用 SendClientAdjustment 打包发出。TimeStamp 告诉客户端"你 T=X 时刻的预测错了"。

客户端：回滚 + 重放 (Replay)
Prediction & Reconciliation 的核心步骤：
  1. 把角色位置直接设为服务端给的 NewLoc
  2. 从 SavedMoves 队列中丢弃 <= TimeStamp 的 Move（服务器已确认）
  3. 重新执行剩余的 SavedMoves（ServerMove_PerformMovement）——玩家在 T~现在这段时间的输入不能白按！
结果：位置抖一下但输入不丢。

阶段 3：SimulatedProxy - 平滑插值

广播属性同步

AActor::ReplicatedMovement (FRepMovement) 包含位置/速度/旋转。Authority 每 NetUpdateFrequency（Character 默认 100Hz）通过 ServerReplicateActors 推给所有 SimulatedProxy。

SmoothClientPosition - 向新位置插值
SimulatedProxy 端不会瞬移到新位置——因为收到的位置是 ~100ms 前的服务端快照，直接跳会很刺眼。

做法：SmoothClientPosition 记录目标位置，每帧向目标插值（NetworkSmoothingMode::Exponential 默认）。

这就是为什么玩家感觉自己零延迟、看别人平滑——两套不同的处理逻辑。

三端总结

端行为关键特点
AutonomousProxy立即执行预测0 延迟感；后台异步等服务端确认；出错回滚+重放->输入不丢
SimulatedProxy无输入权，纯观察收到的是 ~100ms 前的服务端快照；通过插值让"别人"看起来平滑
Authority唯一权威用同样输入验证客户端结果；出错则强制拉回(Reliable RPC)；广播权威位置给所有 SimulatedProxy
