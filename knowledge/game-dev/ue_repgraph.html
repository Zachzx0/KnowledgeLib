UE ReplicationGraph 空间分片架构详解

ReplicationGraph 实现了 UReplicationDriver 接口，接管原生的 ServerReplicateActors。通过空间网格分片、休眠管理、永远相关节点和动态频率等策略，将 CPU 复杂度从 O(N x M) 降到近似 O(k x M)。

Fortnite 100 玩家大逃杀就是靠它扛下来的。

启动时构建 Graph

游戏启动时（InitializeForWorld）构建 Graph 拓扑：创建 GridSpatialization、AlwaysRelevant、Dormancy 等 Node，并通过 AddGlobalGraphNode 挂到根。不同游戏定制自己的 Graph 拓扑（Fortnite 有专门的 UFortReplicationGraph）。

Actor 分类注册

新 Actor Spawn 时，Graph 根据 FClassReplicationInfo（每个类的策略）路由到相应 Node：

  • 静态背景物 -> Node_GridSpatialization2D
  • 玩家 PC -> Node_AlwaysRelevant_ForConnection
  • 场景管理 -> Node_ActorList（全局）
  • 长时间不动 -> Node_DormancyNode

核心 Node 类型

Node_GridSpatialization2D - 空间网格分片

Fortnite 大逃杀的关键节点。把整个 World 切成固定大小的 2D 格子（如 10000x10000），每个 Actor 按位置塞入对应格。Gather 时只查 Viewer 周围 3x3 格，复杂度从 O(N) 降到 O(常数)。

Node_DormancyNode - 休眠管理

处于 DORM_Initial / DORM_DormantAll 的 Actor 停止 poll。有属性变动时游戏代码调 FlushNetDormancy 唤醒。大量静态物件（房屋、装饰）默认 dormant，几乎不占 CPU。

Plugins/Runtime/ReplicationGraph/Source/Public/ReplicationGraph.h:483

Node_AlwaysRelevant_ForConnection - 永远相关

每个 Connection 有独立的一份此 Node。存放：该玩家的 PlayerController、Pawn、PlayerState。无需空间检测，永远列入同步列表。

Node_DynamicSpatialFrequency - 动态频率

根据 Actor 到 Viewer 的距离动态调整 NetUpdateFrequency：近处 30Hz -> 中距 10Hz -> 远处 1Hz。类似 LOD。

核心流程

ServerReplicateActors (重写主循环)

继承自 UReplicationDriver，被 UNetDriver 通过 SetReplicationDriver 挂上。整个 Tick 流程：

  1. Prepare（更新每个 Node）
  2. 遍历每个 Connection
  3. 对每 Connection 调各 Node 的 GatherActorListsForConnection（关键剪枝）
  4. 合并结果 -> 走原生 UActorChannel 发送
Plugins/Runtime/ReplicationGraph/Source/Public/BasicReplicationGraph.h:57

GatherActorListsForConnection - 关键剪枝点

每个 Node 各自实现。传入 FConnectionGatherActorListParameters（Viewer 位置/连接ID/输出列表引用）。GridSpatialization2D 只把 Viewer 周围 3x3 格 Actor 塞入 OutList；DormancyNode 只输出被唤醒的；AlwaysRelevant 无脑输出。

底层复用

RepGraph 只是替换了"选谁"的策略。Gather 出来的候选列表最终仍然：每个 Actor -> 找/创建 UActorChannel -> ReplicateActor() -> FRepLayout::SendProperties -> SendBunch，和原生同步共用一套发包底层。

性能收益：O(N x M) -> O(k x M)

原生：每帧 N 个 Actor x M 个 Connection 的 IsRelevant 检查
RepGraph：每 Connection 只从少数 Node 拿相关列表，k 远小于 N

Fortnite 100 玩家场景（N 约 20000, M=100），CPU 从"服务器炸"降到可接受。

优缺点

  • 优点：CPU 从 O(N x M) 降到接近 O(k x M)，其中 k 远小于 N
  • 缺点：需要为每个游戏定制 Graph 拓扑，学习曲线陡峭
