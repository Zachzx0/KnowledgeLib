UE 原生同步 (Native Replication) 完整数据流

从 UPROPERTY 声明到 UDP 发包的完整链路。核心复杂度 O(N x M)，Fortnite 之前所有 UE 产品都在用它。

1. 开发者接触面

两种方式：UPROPERTY(Replicated) + GetLifetimeReplicatedProps 声明属性同步；UFUNCTION(Server, Reliable/Unreliable) 声明 RPC。UHT 会为每个 UClass 生成 rep-layout 元数据。

2. PreReplication 钩子

服务端每帧调用一次。子步骤：

  1. _PrepConnections 更新每个连接的 Viewer

4. BuildConsiderList

从 NetworkObjects 中筛出：不 Dormant 且 NextUpdateTime 到期的 Actor。这份列表被所有 Connection 共享——但每个连接还要各自算 IsRelevant，这就是 O(N x M) 的来源。

5. PrioritizeActors

为每个 Connection 计算优先级：Actor->NetPriority x (1 / DistanceSq) x TimeSinceLastUpdate。带宽有限时按此排序，只发前 K 个。

6. UActorChannel::ReplicateActor

一个 Actor 一次同步的核心：

  1. 首次同步 -> 序列化 Actor 类信息（生成通道）
  2. 遍历 Actor + 所有 SubObject（Component）
  3. 对每个 Object 调 FObjectReplicator::ReplicateProperties
  4. 用 FRepLayout diff 出改动位
  5. WriteContentBlockPayload 写入 Bunch
  6. SendBunch

7. FObjectReplicator 和 FRepLayout

FObjectReplicator

每个被同步的 UObject（Actor/Component/SubObject）配一个 Replicator，缓存上次发送的 Shadow State。diff 时对比当前值与 Shadow 生成 changed handles。

FRepLayout (反射生成的字段布局)

类初始化时通过反射扫 UPROPERTY(Replicated) 生成 FRepLayoutCmd 序列（每个 Cmd 描述 offset+size+NetSerializeCallback）。

核心方法：

  • DiffProperties - 对比新旧值
  • SendProperties - 写 Bit Stream
  • ReceiveProperties - 读并反序列化
  • SendPropertiesForRPC - RPC 参数序列化

8. FOutBunch -> SendBunch -> FlushNet -> UDP

一个 Bunch = 一次逻辑消息（一次属性更新 / 一个 RPC）。带 header：ChannelIndex/bReliable/bOpen/bClose/PacketID。

多个 Bunch 打包成一个 UDP Packet。可靠 Bunch 存入 OutRec 链表直到 ACK 才释放。大 Bunch 自动切成 PartialBunch。

ACK 是捎带在下一个 Packet 头里的，不是独立包。

9. 客户端收包

客户端 Tick -> UIpNetDriver::TickDispatch -> ReceivedRawPacket -> 解包 header -> 拆出 Bunch -> 按 ChannelIndex 派发到对应 ActorChannel -> ProcessBunch -> FRepLayout::ReceiveProperties 反序列化并直接写内存 -> 触发 OnRep_XXX 回调。

UNetDriver 核心类

  • UNetDriver - 网络驱动主类，管理 ClientConnections，每 Tick 派发 ServerReplicateActors 和 ProcessRemoteFunction
  • UNetConnection - 代表一个客户端连接，持有 ActorChannel 数组、Bunch 打包缓冲、可靠性 ACK 记录
  • UActorChannel - 一个 Actor 一个 Channel，负责 ReplicateActor() 用 FRepLayout diff 后发送
  • FRepLayout - 反射生成的属性布局缓存，负责 diff/序列化/反序列化

性能特点

优点：简单、稳定、Fortnite 之前的所有产品都用它。

缺点：O(N x M) 复杂度，Actor 多、玩家多时 CPU 爆炸。每帧遍历所有 Actor x 所有 Connection 做 IsRelevant 检查和属性 diff。
