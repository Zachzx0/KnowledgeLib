# tutorial-gen Skill

> **版本：** 1.1.0
> **定位：** 把"难入门的主题"转成 15-20 分钟可通关的、自包含的单文件交互式 HTML 教程
> **更新日期：** 2026-05-14

执行目标：`$ARGUMENTS`（来源可以是文件路径 / 目录路径 / URL / 纯文字主题，输出路径可在参数中指定或与用户确认）

---

## 一句话定位

> **原始来源（无论它是什么形式）是教学内容的唯一权威，HTML 是它的"会动的导览"，不超出来源的事实边界。**

---

## 触发场景

- 用户给一份文档/笔记/链接，说"做成新人教程"
- 用户描述一个模块/系统/概念/工具，说"做个交互式入门"
- 用户拿来 research / 第三方文档 / RFC，说"我要给团队讲这个"
- 关键词：`教程`、`html 教程`、`新人入门`、`交互式文档`、`可视化讲解`、`onboarding`、`tech talk`

---

## 输入抽象（4 种来源类型）

| 来源类型                   | 形式                                         | 处理方式                                             |
| -------------------------- | -------------------------------------------- | ---------------------------------------------------- |
| **A. 文本文件**      | `.md` / `.txt` / `.rst` / `.adoc` 等 | 读全文，提取标题层级作为章节候选                     |
| **B. 源代码 / 工程** | 单文件或目录                                 | 先建心智模型（入口、核心类、主流程），再下笔         |
| **C. 网络资源**      | URL（文档站、blog、RFC、wiki）               | 抓取主要内容；大文档先分段总结                       |
| **D. 主题 / 口述**   | 用户用文字描述要讲的主题                     | 反问澄清覆盖范围与已有背景，**不凭空发明**事实 |

**关键判断：来源信息是否足够支撑一份教程？**

- 信息不足（< 300 字纯概念，或代码量 < 200 行但概念多） → **拒绝并请求更多素材**，绝不脑补
- 信息过载（数千行代码 / 多份长文） → 先做"信息蒸馏"，给用户看大纲，确认覆盖范围再开工
- 来源互相冲突（多份资料说法不一） → **先和用户确认以哪个为准**，不要自行调和

## 输出抽象

### 核心产物

- **单个 HTML 文件**（自包含，无外部依赖）
- 文件名约定：`{slug}.html`（slug 由主题导出，e.g. `dsa-system.html`、`trpc-onboarding.html`）

### 输出路径

按以下优先级确定（**不要假设固定路径**）：

1. 用户显式指定（如"输出到 `docs/tutorials/`"）
2. 项目里已存在的 tutorial 目录（探测：`docs/tutorial*/`、`OGDoc/tutorial/`、`tutorials/`、`site/tutorials/` 等）
3. 与原始来源同目录（如 md 在 `docs/` 下 → HTML 也放 `docs/`）
4. 项目根 / 当前工作目录（最后兜底）

**输出前必须与用户确认路径**，除非来源已明确指明。

### 关联资产（可选，按项目情况）

| 资产                   | 是否做 | 触发条件             |
| ---------------------- | ------ | -------------------- |
| 维护 tutorial 索引文件 | 推荐   | 项目已存在类似的索引 |
| 本地预览启动           | 推荐   | 用户希望立即查看     |

## 工作流（6 步）

### Step 0 · 来源理解 + 信息充足度判断

| 来源类型 | 操作                                                             |
| -------- | ---------------------------------------------------------------- |
| 文本文件 | 读全文                                                           |
| 代码目录 | 用 glob / grep 找：入口、关键类、主流程的 RPC/API、配置/环境变量 |
| URL      | 抓取，必要时分段总结                                             |
| 主题     | 与用户对话明确边界                                               |

**结束本步的检查点：**

- ☐ 能用 3 句话讲清楚"这个主题是什么、为什么存在、不是什么"
- ☐ 能列出至少 3 个"新人最容易蒙的点"
- ☐ 能找到至少 1 条值得动画化的"多步骤主流程"

任何一项做不到 → 退回与用户对话。

### Step 1 · 章节大纲设计

按**新人认知顺序**排章节（不要按来源的章节顺序），上限 12 章：

```
1.  一句话理解（What / Why / Not）  ← 抓注意力
2.  全景图（一张图把所有角色摆出来）
3.  核心概念 / 对象（4-6 个，命名 + 类比）
4.  ⭐ 核心流程（最难懂的那条主线，必须做动画）
5.  接口 / 协议 / 输入输出
6.  数据结构 / 状态演示
7.  多变体 / 边缘情形（≤ 4 个）
8.  周期 / 维护 / 守护逻辑
9.  性能 / 降级 / 选项对比
10. 排障 / 常见问题
11. 推荐学习顺序（清单）
12. 自检测验（5 题）
```

**不是所有主题都需要全部 12 章**。简单主题 6-8 章足够。复杂主题如超出 12 章，**拆成多份教程**而非塞一份。

### Step 2 · 章节模式匹配

下面 11 个**交互模式**是已验证的"高频教学单元"，作为**起点参考**而非上限。匹配章节内容到合适的模式，覆盖不到时**鼓励为这章设计新模式**（见后文「创新模式空间」）：

| #  | 模式                            | 何时用                                      | 关键交互                                       | 实现要点                                                           |
| -- | ------------------------------- | ------------------------------------------- | ---------------------------------------------- | ------------------------------------------------------------------ |
| 1  | **概念卡 (concept card)** | 一句话概念 / 角色名片 / What-Why-How 三元组 | 静态直接呈现，所有信息一眼看到                 | 卡片：图标 + 标签 + 标题 + 内容直接铺开；优于翻转卡（见下方备注）  |
| 2  | **可点击 SVG 全景图**     | 系统架构 / 通讯邻居 / 概念地图              | 节点点击 → 详情面板                           | 内联 SVG（不用 mermaid），节点设`data-id`，JS 监听 click         |
| 3  | **⭐ 时序动画播放器**     | 多步骤流程（握手、状态迁移、调用链）        | 上一步 / 下一步 / 自动播放 / 重置 / ← → 键盘 | 数据驱动：`STEPS[]` 数组，每步含 from/to lane、title、desc、code |
| 4  | **双向卡片对照**          | 上下游接口 / 输入输出 / 对比                | 静态                                           | 两个 card 并排，方向用箭头标                                       |
| 5  | **互动查找演示**          | 多索引数据结构 / 路由表 / 字典              | 输入值 → 实时显示命中路径                     | mock 一份典型数据，JS 模拟查找过程                                 |
| 6  | **Tab 切换**              | 同主题的多种变体（≤ 4 个）                 | 顶部 tab 切面板                                | 简单 active 类切换                                                 |
| 7  | **圆形轨道 (orbit)**      | 周期循环 / 巡检 / 定时任务 / 流水线         | 点击轨道上的点查看详情                         | SVG 圆 + 三角函数定位点                                            |
| 8  | **条形图动画**            | 性能 / 资源 / 选项对比                      | 滚动入视图时 width 0 → 目标值                 | `IntersectionObserver` 触发 CSS transition                       |
| 9  | **决策树卡片**            | 排障 / FAQ / 路径选择                       | 点症状卡 → 给排查路径                         | 卡片网格 + 详情面板，data 驱动                                     |
| 10 | **持久化清单**            | 推荐阅读 / 完成度 / 步骤检查                | checkbox + localStorage                        | 用版本化 key 存数组                                                |
| 11 | **选择题测验**            | 章末自检                                    | 单选 + 提交评分 + 解释                         | radio + 评分逻辑 + 解释展开                                        |

> **关于翻转卡（旧 v1.0 主推模式，现降级为可选变体）：**
> 翻转卡（hover / click 翻面显示背面内容）实战效果不佳 —— ① 移动端没有 hover，信息靠隐藏不直观；② 用户可能完全错过背面；③ 增加了一次额外交互成本。**默认用概念卡（信息直接呈现）**。
> 仅在"猜一下答案"的教学场景（如算法面试题、知识点回忆）才考虑翻转，并应在视觉上明显提示"点击翻面"。

**模式匹配示例（不同主题各取所需，仅作起点参考）：**

| 教程主题        | 第 1 章选 | 第 4 章选            | 第 6 章选       | 第 10 章选         |
| --------------- | --------- | -------------------- | --------------- | ------------------ |
| 服务端系统模块  | 概念卡    | 时序动画             | 互动查找        | 决策树             |
| 协议 / RFC 解读 | 概念卡    | 时序动画（握手）     | 双向卡片        | —                 |
| 工具 / CLI 入门 | 概念卡    | Tab 切换（不同命令） | 互动演示        | 决策树（常见错误） |
| 算法 / 数据结构 | 概念卡    | 状态机动画           | 互动查找        | 选择题             |
| 第三方框架      | 概念卡    | 时序动画（生命周期） | Tab（不同 API） | 决策树             |

### Step 3 · 生成 HTML 单文件

**统一骨架：**

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8"/>
  <meta name="viewport" content="width=device-width, initial-scale=1.0"/>
  <title>{title} · 新人交互式教程</title>
  <style>/* 内联 CSS，约 400-600 行 */</style>
</head>
<body>
  <nav class="topnav"><!-- 进度条 + 章节锚点 --></nav>
  <div class="container">
    <header class="hero"><!-- 标题 + 学习路径 stats --></header>
    <section class="chapter" id="ch1"><!-- 模式实例 --></section>
    <!-- ... N 章 ... -->
    <footer><!-- 来源引用 --></footer>
  </div>
  <script>/* 内联 JS，约 300-500 行 */</script>
</body>
</html>
```

**统一 CSS 设计变量（视觉语言锚点）：**

```css
:root {
  --primary: #2563eb;
  --cyan: #06b6d4;
  --green: #10b981;
  --amber: #f59e0b;
  --purple: #8b5cf6;
  --bg-page: #f1f5f9;
  --bg-card: #ffffff;
  --code-bg: #0f172a;
  --code-text: #e2e8f0;
  --border: #e2e8f0;
  --shadow: 0 1px 3px rgba(0,0,0,0.05), 0 4px 12px rgba(15,23,42,0.06);
}
```

**预期文件大小**：50-100 KB。超过 120 KB 通常说明章节超量，回到 Step 1 砍。

### Step 4 · 写入输出位置

**生成 HTML 文件前确认输出路径**（见上文「输出路径」优先级）。

**写入注意事项：**

1. 若 IDE/工具的写文件 API 在该路径受限（例如目录被 `.cursorignore` 屏蔽），用 shell heredoc 写入
2. heredoc 用**带引号**的标签（如 `<<'EOF'`）防止 shell 扩展 `$`, `` ` ``, `\`
3. 文件较大（> 60 KB）时分两次：先写 head + body，再 `>>` 追加 script，避免单次命令 ARG_MAX 撞墙

### Step 5 · 维护索引（可选）

探测项目是否有"教程索引"文件（README.md、index.md、tutorials.md 等），有就追加一行：

```markdown
| [`{name}.html`](./{name}.html) | {来源描述} | {交互元素列表} | {适合人群} |
```

**没有就不要主动新建**。让索引由项目维护者决定何时建立。

### Step 6 · 验证与预览

```bash
# 通用本地预览（任何机器、任何项目都能跑）
cd {output_dir}
python3 -m http.server 8000
# 浏览器打开 http://localhost:8000/{name}.html
```

**SSH 远端场景额外提示**：让用户用 IDE 的端口转发（VS Code / Cursor 自动）或手动 `ssh -L 8000:localhost:8000 host`。

最终给用户：

- 输出文件路径
- 本地预览 URL
- 章节结构表

## 关键设计原则（铁律）

| # | 原则                                                  | 反例                                               |
| - | ----------------------------------------------------- | -------------------------------------------------- |
| 1 | **原始来源是唯一权威**，HTML 不超出来源事实范围 | 在 HTML 里加来源没有的"独家细节"→ 误导 / 知识断层 |
| 2 | **完全自包含**，无 CDN / 外部依赖               | 引 mermaid / d3 CDN → 断网无法看                  |
| 3 | **按新人认知顺序**排章节，不按来源原顺序        | 直接照搬原文 12 个 section → 第一章就是"对象列表" |
| 4 | **每章 1-3 屏**，超过就拆                       | 单章 15 屏滚屏 → 没人看得完                       |
| 5 | **交互必须有教学价值**，不为动效而动效          | 标题闪烁、滚动视差 → 干扰理解                     |
| 6 | **核心流程必须做动画**（模式 3）                | 时序图截个静态图 → 浪费 HTML 优势                 |
| 7 | **章末必有"标记完成" + localStorage**           | 没进度 → 教程没"通关感"                           |
| 8 | **不假设环境**（项目结构、目录约定、工具链）    | 硬编码路径 / 引用项目特有机制                      |

## 章节模式实现要点速查

### 模式 1 · 概念卡（信息直接呈现，无需翻转）

每张卡片正面直接铺开**全部信息**，新人扫一眼就抓到要点：

```html
<div class="concept-card">
  <div class="icon">🎯</div>        <!-- 单个图标，仅作章节标识 -->
  <div class="label">WHAT / WHY / NOT</div>   <!-- 类别标签（一两词）-->
  <div class="title">{核心标题}</div>
  <div class="content">{1-2 句话内容，避免长段落}</div>
</div>
```

**多张卡片组合用 grid**（`repeat(auto-fit, minmax(220px, 1fr))`），三栏并列，桌面端 / 移动端都一目了然。

### 模式 3 · 时序动画播放器（最关键，必须做好）

数据驱动，每步必填这 5 个字段：

```javascript
const STEPS = [
  {
    from: 'LaneA', to: 'LaneB',   // 泳道 id
    y: 80,                         // SVG y 坐标
    msg: 'API 名 / 事件名',         // 箭头上的文字
    title: '人话标题',
    desc: '为什么这步重要（HTML 允许）',
    code: '代码位置 / 关键参数（允许语法高亮 span）'
  },
];
```

必须支持的交互：**上一步 / 下一步 / 自动播放 / 重置 / 键盘 ← →**。

### 模式 10 · 持久化清单

```javascript
const STORE_KEY = 'tut_{slug}_v1';  // 永远加版本号
function loadState() { /* 从 localStorage 读 */ }
function saveState() { /* 任何变化都 save */ }
```

**版本号后缀** 是为了未来重构教程结构时不与旧数据冲突。

### 模式 5 · 互动查找演示

```javascript
const MOCK_DATA = { /* 一份典型数据 */ };
const LOOKUP_DESC = {
  // 每个可查询字段配一个"路径说明 + 用途"
};
// 用户输入 → 命中 / 未命中 → 显示完整查找路径
```

## 创新模式空间（鼓励为特定主题设计新交互）

11 个已验证模式是高频"教学单元"的归纳，**不是设计上限**。当一章的内容用现有模式呈现得别扭、或者主题本身有独特的交互可能性时，**鼓励**设计新模式。

### 何时考虑新模式

- 现有模式套上去后**信息密度反而下降**（要绕弯子才能讲清楚）
- 主题本身有独特的"动起来更直观"的特性（如：物理引擎演示、数据流可视化、密码学步骤）
- 已有模式不能体现的**对比 / 演化 / 因果**关系

### 新模式的接受标准

设计新模式时同时满足：

```
□ 教学价值明确：能用 1 句话回答"这个交互让新人理解了什么"
□ 视觉简洁：不超出整套教程的视觉语言（沿用 CSS 变量）
□ 自包含：不引入 CDN / 外部库 / 任何网络请求
□ 实现成本可控：建议 ≤ 100 行 JS + ≤ 50 行 HTML
□ 不破坏整体认知节奏：单次交互完成时间 ≤ 30 秒
□ 移动端可用：响应式 / 触屏可操作
□ 可访问性兜底：键盘也能操作或至少能 fallback 到静态展示
```

### 新模式示例（受启发但尚未沉淀进 11 模式库）

| 候选模式           | 适用场景                                 |
| ------------------ | ---------------------------------------- |
| 拖拽排序           | "把以下事件按发生顺序排列" 类型自检      |
| Code diff 高亮播放 | 演示重构 / Bug 修复 / 协议演化           |
| 双轴时间轴         | 多服务并发时序（比单泳道时序图维度高）   |
| Mini REPL          | 工具 / DSL 入门，让读者在页内试一行命令  |
| 拓扑变形           | 演示集群伸缩 / 故障切换 / 数据分片再平衡 |
| 数据流动画         | ETL / 流处理 / 消息队列扇出扇入          |

发明新模式后建议：① 在本教程内打磨好；② 用过 ≥ 2 次后，反向沉淀回这份 skill 的 11 模式表。

## 来源类型适配技巧

### 文本文件 → HTML

- 标题层级 ≈ 章节候选，但**不要 1:1 映射**
- 原文表格保留 → HTML 表格；原文列表 → 卡片或清单
- 原文流程图（mermaid / ASCII / 图片）→ 用模式 2 / 3 / 7 重新画

### 源代码 → HTML

1. 先做"心智模型"提取：3 个核心类 + 1-2 条主流程
2. 入口文件作为模式 2（架构图）的中心节点
3. 主流程做模式 3（动画）
4. 类字段 → 模式 5（互动查找）的 MOCK_DATA
5. 状态机 / 生命周期 → 模式 3 或模式 7

### 网络资源 / 第三方文档 → HTML

- 先抓取全文 + 抽取关键概念
- **footer 必须注明来源 URL**，尊重原作
- 章节顺序优先按"新人接触顺序"重组，不照搬原文
- 若原文超长 → 选取核心子集做教程，不要囫囵吞

### 主题 / 口述 → HTML

- 先与用户对话确认：覆盖范围、读者背景、关键术语
- 提炼 3-5 个"反直觉发现" → 章节 1 用模式 1（概念卡，每张卡直接铺开"发现 + 一句话解释"）
- 假设 vs 结论 → 模式 6（Tab）
- 数据 / 实验结果（若有）→ 模式 8（条形图）

### Research 笔记 → HTML

- 笔记往往非线性 → 强制重组为线性章节
- 核心 insight → 模式 1（概念卡）放前面，每条 insight 一张卡
- 论据 / 数据 → 模式 8（条形图）/ 模式 5（互动查找）
- 开放问题 → 章末用 callout 形式列出，**不强行给答案**

## 教程库维护（多教程场景）

当输出目录积累到 ≥ 3 份教程时，需要从"单文件思维"切换到"教程库思维"。以下是建议（不强制，按团队选取）：

### 1. 索引页（强烈建议）

在教程目录根放一个 `index.html`，作为统一入口。这是一个**特殊教程**，本身用模式 1 + 模式 9 实现：

| 内容                                         | 用模式                                                |
| -------------------------------------------- | ----------------------------------------------------- |
| 教程库简介                                   | 简单 hero                                             |
| 全部教程卡片网格                             | 模式 9 决策树卡片（点卡片 → 跳转，外加学习路径推荐） |
| 按主题分类筛选                               | 简单 tag filter（可选）                               |
| 学习路径（"想成为 X，按顺序读 A → B → C"） | 模式 10 持久化清单                                    |

每张教程卡片建议字段：

```
- 标题
- 一句话介绍
- 标签（领域 / 难度 / 受众）
- 预计时长
- 最后更新日期
- 完成度（来自 localStorage，显示进度条）
```

**index.html 是动态生成还是手维护？** 建议保留一份 `INDEX.json` 数据源（手维护），由模板生成 `index.html`。这样新增教程时只改 JSON。

### 2. 命名约定

- **slug 用 kebab-case**：`dsa-system.html` ✓，`DSASystem.html` ✗，`dsa_system.html` ✗
- **slug 与主题强相关**，避免 `tutorial-01.html` 这类无信息编号
- **多版本场景用日期后缀**：`oauth2-basics.html` → `oauth2-basics-v2.html`（而非覆盖原文件，让旧链接仍有效）

### 3. 元数据（在 HTML 内嵌）

每个教程在 `<head>` 区放标准 meta 标签，便于库维护脚本扫描：

```html
<meta name="tutorial:source" content="docs/foo.md">
<meta name="tutorial:source-mtime" content="2026-05-14">
<meta name="tutorial:generated" content="2026-05-14">
<meta name="tutorial:topics" content="auth,security,oauth2">
<meta name="tutorial:audience" content="backend-newcomer">
<meta name="tutorial:duration-min" content="20">
<meta name="tutorial:status" content="published">  <!-- draft / published / deprecated -->
```

### 4. 过期检测（lint 脚本，可选）

教程内容必须跟原始来源同步。简单 lint 思路：

```
for each *.html in tutorial-dir:
    src = read meta "tutorial:source"
    if src is local file:
        src_mtime = stat(src).mtime
        gen_mtime = read meta "tutorial:source-mtime"
        if src_mtime > gen_mtime:
            warn "教程 {file} 的来源已更新（{src_mtime}），建议重新生成"
```

15 行 Python 即可。集成到 CI / pre-commit 后，**来源改了但教程没更新会被拦下**。

### 5. 视觉一致性

每个教程依然**完全自包含**（不抽公共 CSS 文件，避免依赖断链），但共享同一套 **CSS 变量**：

```css
:root {
  --primary: #2563eb;
  --cyan: #06b6d4;
  /* ... 见 Step 3 的统一 CSS 变量 */
}
```

把这套变量放进一个**模板文件** `_template.html` 留在目录里作参考，生成新教程时复制粘贴。**禁止用 `<link rel="stylesheet">` 引入公共 CSS**——破坏自包含。

### 6. 教程间的关联

如果教程之间有"前置 / 后续"关系（如"先读 protocol-basics 再读 oauth2-advanced"），在教程顶部 hero 区加一个**前置卡片**：

```html
<div class="prereq-card">
  <span class="label">前置阅读</span>
  <a href="protocol-basics.html">📖 协议基础</a>
</div>
```

并在 hero 后加 **"通关后推荐"** 链接卡。

### 7. 生命周期管理

| 状态 | meta status 值               | 处理                                         |
| ---- | ---------------------------- | -------------------------------------------- |
| 草稿 | `draft`                    | 不在 index.html 显示 / 显示但加 "Draft" 标签 |
| 发布 | `published`                | 正常显示                                     |
| 废弃 | `deprecated`               | index.html 灰显 + 跳出"已废弃"提示           |
| 归档 | （移到`_archive/` 子目录） | 从主索引移除，旧 URL 仍可访问                |

### 8. 目录布局建议

```
{tutorial-dir}/
├── index.html             # 主入口（导航 + 学习路径）
├── INDEX.json             # 索引数据源（手维护或脚本生成）
├── _template.html         # 新教程的模板（含 CSS 变量、骨架）
├── _check.py              # 过期检测 lint（可选）
├── README.md              # 教程库使用说明（给人和 AI）
├── {topic-a}.html         # 一份教程一文件
├── {topic-b}.html
├── ...
└── _archive/              # 归档（旧版本 / 已废弃，URL 仍有效）
    └── {old-topic}.html
```

下划线前缀 `_template`、`_check`、`_archive` 表示"不是教程本体"，方便 lint 脚本与索引扫描时跳过。

## 集成检查清单（每次生成完跑一遍）

```
□ 文件已写入与用户确认的输出路径
□ 输出路径不是硬编码（如果是新项目，已与用户对齐）
□ 文件大小 50-100 KB
□ 章节数 ≤ 12，每章 1-3 屏
□ 至少包含模式 3（时序动画）或模式 2（可点击全景）—— HTML > 原文档的核心理由
□ 全部内联 CSS + JS，grep 整个文件 'http' / 'https' 应只命中说明性链接（footer 的来源引用）
□ 顶部导航 + 进度条 + localStorage 持久化俱全
□ 章末有"标记完成"按钮
□ 自检测验 ≥ 5 题，覆盖前 4 章核心概念
□ Footer 注明原始来源（文件路径 / URL / "来自用户口述"）
□ 移动端响应式（至少 1 处 @media (max-width: 720px) 适配）
□ 已在 <head> 加入 tutorial:* meta 标签（source / source-mtime / topics / status 等）
□ 若项目有教程索引文件，已追加条目
□ 若教程目录已 ≥ 3 份，提醒用户考虑建立 index.html / INDEX.json
□ 本地预览验证 200 OK（或给出明确的预览命令）
```

## 明确不做

- ❌ **不超出原始来源的事实范围**（不脑补、不编造、不调和冲突信息）
- ❌ **不硬编码项目路径 / 目录约定**（不假设有 `docs/`  / `tutorials/` 等）
- ❌ **不假设特定 IDE / 工具链**（不写 `.cursorignore` / `.vscode` 等独有机制为必需步骤）
- ❌ **不超过 12 章**（认知负载，超出就拆成多份教程）
- ❌ **不在 chat 输出完整 HTML 内容**（太长，告诉用户文件路径与预览 URL 即可）
- ❌ **不替代原始来源**（教程是来源的"可视化分发"，不是新事实源）
