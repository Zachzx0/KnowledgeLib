1.1 两台机器，两个宇宙

玩家 A 在东京按下扳机，玩家 B 在纽约看到的应该是同一个弹道。但物理上，A 和 B 跑的是两份独立的游戏进程——两台机器各自在自己的内存里模拟着一个"平行宇宙"。 网络同步的使命，就是让这两个平行宇宙的关键物体保持一致，同时承受互联网这条"不可靠的太平洋"。

同步的三个基本问题
  1. 谁说了算（Authority）？——冲突时以谁为准
  2. 什么要同步（State）？——哪些变量跨机传递
  3. 多久同步一次（Tick Rate）？——频率与带宽的权衡

1.3 状态同步 vs 帧同步：你的游戏更适合哪个？

State Sync · 状态同步

权威端（通常是服务器）跑完整模拟，定期把"谁在哪/血量多少/子弹剩几颗"这些状态快照广播出去。 其它机器按快照插值/外推显示。

优点 容忍丢包 / 不需要确定性 / 开发友好
缺点 带宽随"可见对象 × 连接数"上涨
典型 FPS / MOBA / MMO / 开放世界 · UE 的默认路径，Iris 也走这条
Lockstep · 帧同步

只同步玩家输入，所有机器在相同逻辑帧上跑确定性模拟，自然得到一致结果。

优点 带宽极低 / 回放便宜 / 公平
缺点 要求严格确定性 / 对延迟敏感 / 反作弊脆弱
典型 RTS / 格斗 / 自走棋 · UE 原生复制不做这个

✅ 自检

Q1. 为什么 FPS/MOBA 几乎都选状态同步？
FPS/MOBA 对延迟敏感、物理不一定确定、反作弊要求高；服务器权威的状态同步能做"客户端预测 + 服务器回滚"，且不依赖跨平台浮点确定性。
Q2. 100ms RTT，服务器 30Hz，客户端大约滞后多少帧？
单程 ~50ms ≈ 1.5 个 Server Tick；再加插值缓冲 ~100ms，通常客户端看到的是 3~5 帧前的状态。
Q3. 为什么说"带宽是 Iris 的主战场"？
RTT 基本取决于物理距离，丢包很大程度靠运营商；这两个工程师能动的空间小。但"每帧传多少字节"——量化、脏位、过滤、优先级——是引擎可控的主要旋钮，Iris 的绝大部分设计都在这里下刀。

2.3 ENetRole：我这个 Actor 是谁？

同一个 Actor 在不同进程里扮演不同角色。每个 Actor 有一对 Role（本端角色）和 RemoteRole（远端角色），两者在"本端 / 远端"之间是镜像翻转的。

Server 视角 Authority

Role = ROLE_Authority
RemoteRole = ROLE_AutonomousProxy
自己的 Client AutonomousProxy
我正在控制它，能做客户端预测、发 Server RPC。
Role = ROLE_AutonomousProxy
RemoteRole = ROLE_Authority
别人的 Client SimulatedProxy

RemoteRole = ROLE_Authority
易错点：判断"能不能改这个属性"请用 HasAuthority()，不要写 Role == ROLE_Authority——前者在 Standalone 下也成立。

2.4 Driver / Connection / Channel：一家三代

  • NetDriver 每帧在 TickFlush 里干两件事：把入站包分发给各 Connection，再让各 Connection 决定这帧要发什么（Legacy：ServerReplicateActors；Iris：UReplicationSystem::PreSendUpdate）。
  • NetConnection 是一端玩家的物理承载，负责拼 Packet、跑 Ack/Nak、算带宽预算。
  • Channel 是逻辑通道：UControlChannel（登录/控制）/ UActorChannel（Legacy 复制主战场）/ UVoiceChannel（语音）。Iris 启用后，属性复制不再走 ActorChannel，而是通过 DataStream 共享一条管道。

✅ 自检

Q1. Listen Server 下，主机玩家自己的角色 Role 是什么？
ROLE_Authority。因为 Listen Server 进程同时是 Server，也是这个玩家的本地端，权威就在本地。
Q2. 为什么判断权威要用 HasAuthority() 而不是 Role == ROLE_Authority？
两者在 Standalone 下都为真；但使用 HasAuthority() 更能表达意图，也能被引擎在未来重构下保持语义。
Q3. FNetGUID 和 FNetRefHandle 能互换吗？
不能。两者是并行体系：Legacy 的复制通路用前者，Iris 的用后者。但 Iris 启用时仍能通过 IrisObjectReferenceCache 把本地 UObject 索引回来。

✅ 自检

Q1. Reliable RPC 会阻塞后续 RPC 吗？
同一个 Actor Channel 上的 Reliable RPC 是按序发送的；丢包重发期间，同通道后续 Reliable Bunch 会等它确认。这就是为什么"大量 Reliable RPC 挤爆一个 Actor"是常见性能事故。
Q2. DORM_Initial 与 DORM_DormantAll 差在哪？
DORM_Initial：首次初始状态复制完就睡；DORM_DormantAll：持续保持休眠，需显式 FlushNetDormancy 才会再次复制。前者适合"一次性发一次"的静态地形道具，后者适合"偶尔才变"的可激活物件。
Q3. 为什么 Legacy 下同一个 Actor 在每个连接都要存一份 Changelist？
因为每个 Client 的 Ack 进度不同——必须记住"上一次成功发给你的是什么、从那以后又变了哪些"。Iris 用共享基线（DeltaCompressionBaselineManager）+ 每连接位图把这份开销砍成只剩"位图 + 指针"。

③ 单线程瓶颈

UNetDriver::ServerReplicateActors 全程跑在主线程，大量方法虚函数（IsNetRelevantFor、PreReplication、NetSerialize）直接回调到游戏代码， 要把它并行化几乎等于"重写一半引擎 API"。Ch10 的 Iris PreSendUpdate 流水线在这里下了完全相反的赌注——用 DOD 重构数据布局，让 Filter / Quantize / Write 都能变成 Job。

④ Push Model "不彻底"

MARK_PROPERTY_DIRTY_FROM_NAME 从 UE 4.26 开始让你告诉引擎"这个属性真的变了"， 但 Legacy 的路径是 "Push Model 打脏 → 还是进 CompareProperties 走一遍"——只是把"要不要比"的判定提前，最终仍然依赖 memcmp 兜底，且维护成本很高（每个 UPROPERTY 都要手写 MARK_PROPERTY_DIRTY）。

Iris 的反转：把 Push Model 升格为一等公民—— FDirtyNetObjectTracker + ChangeMask 位图直接是"真相"，轮询 CopyDirtyStateFromInstances 只在未启用 Push Model 的旧 USTRUCT 上做兜底。 详细见 Ch10 流水线的 Stage 5~6。

⑤ 量化碎片化

Legacy 的 NetSerialize 被设计成"每个 USTRUCT 自己实现一段"。于是：

  • FVector_NetQuantize / FVector_NetQuantize10 / FVector_NetQuantize100——同一个 FVector 有四五种量化方案，各自 copy-paste
  • 没有统一的"量化中间态"概念，Quantize 结果直接写 Bit Stream，无法被上层 Delta 模块复用
  • 新增一个 NetSerializer 要同时修改 IsNetRelevantFor / RepLayout / PushModel 多处配合

✅ 自检

Q1. Per-Connection 冗余是"架构性"还是"实现性"痛点？为什么？
架构性。因为 FObjectReplicator 的分配粒度就是 (Actor, Connection)——想去掉 N 份副本，必须把"每连接独立 Changelist"这个前提打破。Iris 通过共享 Protocol + 每连接位图实现。
Q2. Legacy 里有 Push Model，为什么还要做兜底比较？
因为并非所有属性都接入了 Push Model，而且子对象（SubObject）、FastArray、嵌套 USTRUCT 路径上仍可能"绕过" MARK_PROPERTY_DIRTY。兜底比较就是"即使你漏打脏，引擎也能发现"。
Q3. 从 4v4 升到 100v100，Legacy 的复制 CPU 大致增长几倍？
瓶颈在 O(Actor × Conn)：连接数 ×12.5、动态 Actor 数 ×10，所以基础复制比较开销约 ×125。加上带宽争用、Relevancy 检查本身也是 N×M，实际更严重——这正是 Ep 决定重写的触发点之一。

① 共享状态（Shared Instance State）

核心主张：一个 Actor 在服务器内存里的"复制相关的状态"只存一份。每个连接只记一串 位图： "我已经收到到哪个基线（Baseline）了、从那以后变脏了哪些字段"。

  • 实现载体：FReplicationInstanceProtocol（描述结构）+ FReplicationInstanceState（实际状态）+ 每连接 FNetBitArray
  • 配套基础：FDeltaCompressionBaselineManager 做"共享基线"的滚动回收
  • 解痛点 ①：元数据从 O(Actor × Conn) → O(Actor) + O(Conn × 脏位)

② Protocol 驱动（Protocol-driven）

核心主张：每个"可复制类型"对应一个只读、可跨对象共享的 FReplicationProtocol， 把"字段列表 / 偏移 / NetSerializer / ChangeMask 索引"全都编译期烘焙成一张表。运行时的 Write / Read / Compare / Diff 全都"查表"，无虚函数。

  • Protocol 是纯数据：1 份描述符 → N 个实例共享 → 极大降低 I-cache 压力
  • 新加类型只需实现一套 NetSerializer 合同（Quantize / Serialize / Deserialize / Dequantize / IsEqual 等）
  • 解痛点 ⑤：替换"每个 NetSerialize 各写各的"碎片化方案

③ Push Model First-Class

核心主张：脏位图 = 唯一真相。没有 Push 打脏 → 就不会进入 Write。 FDirtyNetObjectTracker 在游戏线程提交，CopyDirtyStateFromInstances 只负责把已打脏对象的实例值"抓拍"到量化缓冲。

  • Legacy 的 MARK_PROPERTY_DIRTY 宏通过 LegacyPushModel.h 桥接到 Iris 的 ChangeMask，老代码无痛过渡
  • 完全避免了 O(N×M) 的 memcmp，只有真正变化的字段会被 touch
  • 解痛点 ② + ④

④ Data-Oriented Design（DOD）

核心主张：把复制相关的状态按"字段而非对象"布局（SoA, Structure of Arrays），让 Filter / Quantize / Write 三个阶段变成"对连续数组做批处理"。

  • 典型数据结构：FNetBitArray（按位并行操作）、TNetArrayView（连续内存视图）
  • 最大收益：可 SIMD / 可 Job 化——这是目标 ⑤ 的物理前提
  • 解痛点 ③（以"数据可并行"的方式解，而不是"加线程锁"的方式解）

⑤ 可并行、分阶段的流水线

核心主张：把每帧复制切成 14 个明确的阶段，每个阶段的输入/输出都是纯数据数组，阶段之间无共享可变状态。 游戏线程只负责调度，实际工作交给 Task Graph。

  • 入口：UReplicationSystem::PreSendUpdate / SendUpdate
  • Ch10 将用一个交互动画逐阶段拆解
  • 解痛点 ③（调度层面的并行化）

✅ 自检

Q1. 为什么说"DOD 是并行化的前提"而不是"并行化是 DOD 的后果"？
传统 OO 布局下，复制状态分散在各 Actor 的成员里，字段对齐、虚函数、缓存局部性都不利。只有先把数据重新编排成"字段连续、无指针间接"的形状，才能真正让 Job 无锁并行——这是物理前提，而非锦上添花。
Q2. Protocol 驱动和 Push Model 为什么必须一起出现？
Push Model 提供"哪些字段脏了"，但需要知道"字段叫什么、多宽、怎么量化"——这是 Protocol 提供的。没有 Protocol，脏位图就只是一串数字；没有 Push Model，Protocol 还得挂一套兜底比较。
Q3. Iris 声称"无虚函数"，那 NetSerializer 不是接口吗？
Iris 的 NetSerializer 是"结构体 + 静态函数指针表"（FNetSerializer 结构），不是虚类。通过宏 UE_NET_IMPLEMENT_SERIALIZER 烘焙出来的全是函数指针，调用点是间接 call，不是 vtable。

7.1 FNetRefHandle · 64 位身份证

真实字段布局（摘自 NetRefHandle.h）：

class FNetRefHandle
{
    // ...
    union {
        struct {
            uint64 Id : 60;                  // 最低位：1 = Static（关卡对象），0 = Dynamic（运行时 Spawn）
            uint64 ReplicationSystemId : 4;  // 一个 PIE / 编辑器下支持最多 16 个 ReplicationSystem
        };
        uint64 Value;
    };
};
  • Static 对象（关卡内预先摆放、有稳定 PathName）：服务器和客户端用同一 Id 约定好，无需发"我是谁"这条协议。
  • Dynamic 对象（运行时 Spawn）：Id 由 FNetRefHandleManager::AllocateNetRefHandle 分配，首次复制时随 Bridge 握手下发给客户端。
  • ReplicationSystemId：PIE 下同时跑 Server + 多个 Client 模拟时，每个 UReplicationSystem 实例都有独立 Id 避免句柄混淆。
  • 易混：Iris 的 FNetRefHandle ≠ Core 的 FNetHandle；后者是更抽象的跨系统句柄（Net/Core/NetHandle/NetHandle.h），Iris 在 FReplicationView 等跨系统接口上用它，而在 Iris 内部用 FNetRefHandle。

7.2 Protocol + Descriptor · 类型描述

Protocol 是某个 UClass（或 USTRUCT）"怎么复制"的总表，Iris 启动时由 FReplicationProtocolManager 一次性烘焙； 它内部持有一个 Descriptor 列表，每个 Descriptor 描述一段"State"（一组共享同一份内存/同一块 ChangeMask 的成员变量）。

  • Descriptor 里 连续的 ChangeMask 位一起描述某段 State 的脏位——Ch10 流水线会直接读这段位做"这帧要不要 Write"。
  • 整个 Protocol 是只读 + 可共享的：1000 个 AFoo 共用一个 Protocol 指针，I-Cache / 内存占用双赢。

7.3 Fragment · 对象与描述符之间的胶水

Fragment 是"让 UObject 能说出自己要复制什么"的单位。每个 Fragment 绑定一个或多个 ReplicationState，并提供两件事：

  • 服务端：从对象拉状态进量化缓冲（Poll / 自动 Push）
  • 客户端：把量化缓冲的结果应用回对象（ApplyReplicatedState）并触发 RepNotify

源码签名（ReplicationFragment.h）：

class FReplicationFragment
{
public:
    explicit FReplicationFragment(EReplicationFragmentTraits InTraits);
    virtual ~FReplicationFragment() {}

    virtual void ApplyReplicatedState(FReplicationStateApplyContext&) const = 0;   // 必须实现：客户端应用
    virtual void CollectOwner(FReplicationStateOwnerCollector*) const {}             // 兼容：报告 outer
    virtual void CallRepNotifies(FReplicationStateApplyContext&) {}                  // 兼容：触发 OnRep_
    virtual bool PollReplicatedState(EReplicationFragmentPollFlags) { return false; }// 未接 Push 时兜底轮询

protected:
    EReplicationFragmentTraits Traits;
};

Traits 位域列出了 Fragment 的"能力声明"，影响它在流水线哪些阶段被访问。重点掌握这 6 个：

Trait含义什么时候会被置位
CanReplicate可作为"数据源"发送几乎所有 Fragment
CanReceive可作为"数据汇"接收客户端路径
HasPushBasedDirtiness走真正的 Push Model，不用轮询接入 MARK_PROPERTY_DIRTY 的 UPROPERTY
NeedsPoll需要流水线兜底轮询没接 Push 的老 USTRUCT
HasRepNotifies含 OnRep_* 回调任何 UPROPERTY 带 ReplicatedUsing
HasObjectReference含跨对象引用，需解析属性类型为 TObjectPtr / UObject*

Fragment源文件用途典型触发

Q1. 1000 个相同 UClass 的对象，Protocol / Descriptor / InstanceState 各存几份？

8.3 NetBlob · "一次性载荷"的统一抽象

先把边界划清楚（重要）
Iris 里并不是"所有网络数据都是 NetBlob"。属性状态走的是独立的 State Stream（由 Protocol 直接驱动 NetSerializer 写主 BitStream），不经过 FNetBlob。
FNetBlob 只装一次性载荷——即下列三类：
  1. RPC（FNetRPCBlob / 派生自 FNetObjectAttachment）
  2. 自定义 Attachment（用户自己实现的 UNetBlobHandler）
  3. 通用拆包载体 FPartialNetBlob——任何 NetBlob/Attachment 体积超阈值（Reliable 默认 192 B），或对象状态过大走 HugeObject 路径时产出。不是 HugeObject 专属，详见 §13.5

所以 FNetBlob 是 Legacy "Bunch" 的精神继承，但只替代了 Bunch 的一部分：RPC / Attachment 这部分。属性复制 Bunch 的职责被 FReplicationProtocol + ChangeMask 直接吸收掉了——不再需要"包裹"。

8.4 NetToken · 跨连接字典

Q1. 为什么 Quantize 必须产出 POD？
因为 Quantized 结果要被 memcpy 进连续缓冲、参与 IsEqual 比较、参与 Delta 基线计算——任何带构造/析构/指针的类型都不安全，且会破坏 DOD 的批处理前提。
Q2. 属性复制和 RPC 共享 DataStream，它们怎么不互相阻塞？
DataStreamManager 对每条流有独立带宽预算与独立读写游标；一条流阻塞不影响另一条的编解码。真正"阻塞"只发生在 UDP 丢包重发层面，那是 NetConnection 的事。
Q3. FNameTokenStore 和 UE Core 的 FName 表是一回事吗？
不是。FName 是本进程内的全局表，Iris 的 NameTokenStore 是"跨连接同步的、按使用需要下发的"子表；它引用 FName，但有自己的序列号与分发协议。

Q1. 一个 Connection 能有多个 View 吗？什么场景会用到？
能。分屏合作（两名玩家共用一个网络连接，各自一个 View）、观战（Spectator 跟随多个目标）、VR 辅助视角都是。默认 4 个 inline，超出动态分配。

10.1 为什么"一帧"要切成 14 段

Legacy 的做法是"一个循环套一个循环"：外层遍历 Connection，内层遍历 Actor，每个 Actor 在循环里一气呵成做完 Relevancy + Compare + Serialize + Write。这段代码耦合极重，无法并行，也很难插 Profile 点。

Iris 的思路正相反——把一帧切成 14 段单一职责的阶段，每段只读 / 只写固定几张表。这样：

  • ① 每一阶段都能独立 Profile、独立开关（Net.Iris.* CVars）；
  • ② 单阶段内部可以放心开 ParallelFor（对象间无共享写入）；
  • ③ 每连接 Writer 只在 Stage 10+ 才被触碰，避免重复 O(N·M) 比较；
  • ④ Ch04 的五大痛点恰好分布在不同阶段，可按图索骥。

11.3 Reader · 为什么只有三态

客户端 FReplicationReader 不管 ACK，它只做两件事：把字节翻译成状态，然后分派给 Bridge。它的"状态"只决定"现在应该调哪条 Dispatch 路径"：

stateDiagram-v2 [*] --> WaitingForCreate WaitingForCreate --> Created: 收到 CreationInfo，调 BeginInstantiateFromRemote Created --> Created: 收到 StateData，走 Deserialize + DispatchStateData Created --> Destroying: 收到 Destroy 标记 Destroying --> [*]: 调 DetachInstance，释放内部槽

Reader 还会把"引用尚未就绪"（UnresolvedReferences）的对象挂起，等 Ch08 的 IrisObjectReferenceCache 解析回来后再一次性 DispatchStateData——这解决了 Legacy 的 "GUID 延后" 痛点。

Q1. 为什么 WaitOnCreateConfirmation 阶段仍可发送 State？
→ 节流考虑。等 ACK 期间字段可能又变了，允许继续发，但发时必须捎带创建信息（防止对端只收到"尾部"时无法解码）。

⏤ FPartialNetBlob 是通用拆包机制（≠ HugeObject 专属）
勘误（与早期版本相比）
早期版本（包括本教程 §13.6 旧图）把 FPartialNetBlob 描述为"HugeObject 专属"或"属性借道 Blob 通道的唯一交汇点"——这是错的。 源码里 FPartialNetBlob 是通用拆包载体，与 HugeObject 不绑定。下面列出三条独立触发路径，每条都附源码出处。

3 条独立触发路径，全部产出 FPartialNetBlob：

  1. 每个 Attachment 入队前的预检（最常见）—— FNetBlobManager::ProcessQueue 对队列里每一个新提交的 RPC / 自定义 Attachment 调用 UPartialNetObjectAttachmentHandler::PreSerializeAndSplitNetBlob：
    • 试写到一块 BitCountSplitThreshold 大小的临时缓冲
    • 溢出 → 走 FPartialNetBlob::SplitNetBlob 拆成多块；不溢出 → 包成 FShrinkWrapNetObjectAttachment 一整块入队
    • 默认阈值（来自 UPartialNetObjectAttachmentHandlerConfig）：Reliable 192 B · Server Unreliable 256 B · Client Unreliable 850 B
    源码：PartialNetObjectAttachmentHandler.cpp::PreSerializeAndSplitNetBlob，日志 tag [RPC_PRE_SERIALIZE_SPLIT]。也就是说 Reliable RPC payload 只要超过 192 字节就会被拆，根本不需要"巨型对象"。
  2. 单对象状态过大（HugeObject 路径）—— Writer 在 PrepareAndSendHugeObjectPayload 里把整段量化 state 写到一块大缓冲（默认上限由 TotalMaxPayloadBitCount = MaxPartBitCount × MaxPartCount 决定，缺省 128 B × 4096 ≈ 512 KB），再用 FRawDataNetBlob 包装 → SplitRawDataNetBlob → 一组 FPartialNetBlob，挂到特殊索引 ObjectIndexForOOBAttachment 上跨帧续写。
    源码：ReplicationWriter.cpp::PrepareAndSendHugeObjectPayload + PartialNetObjectAttachmentHandler.cpp::SplitRawDataNetBlob。这才是 Ch11 Writer 状态机里 HugeObject 特殊态的本意。
  3. Attachment 累积写不下时的回退—— Writer 主循环里若 attachment 子流空间足够大却仍写不下当前批次，会 DoOverflow() 回滚，整批 attachment 转 HugeObject 通道重发： 
    // ReplicationWriter.cpp · ~3253
const uint32 SplitThreshold = PartialNetObjectAttachmentHandler->GetConfig()->GetBitCountSplitThreshold() * 2;
const bool bFallbackToHugeObjectPath =
    BatchEntry.AttachmentType != ENetObjectAttachmentType::HugeObject
    && BatchEntry.bHasUnsentAttachments
    && (BitsThatWasAvailableForAttachements >= SplitThreshold);
if (bFallbackToHugeObjectPath) { Writer.DoOverflow(); /* → 转 HugeObject 路径重试 */ }
    日志 tag "forcing fallback on hugeobject for attachments"。

所以： FPartialNetBlob 是由阈值驱动的通用拆包机制，三类源头都可能产出它：①常规大 RPC ②单对象 HugeState ③ attachment 批量回退。 属性状态仍然不出现在 FNetBlob 里——除了它通过 ②/③ 走 HugeObject 通道的特殊情形。

⏤ 所以 NetBlob 的"统一"指的是什么

Legacy 时 RPC 走 UActorChannel::SendBunch，其他自定义消息各写各的——每种一次性载荷都发明一套序列化协议。Iris 的解法是把"一次性载荷"这个概念统一成 FNetBlob：

  • §11 ProcessNetObjectAttachmentSendQueue 同一段代码就能分发 RPC、Attachment、HugeObject 分片；
  • 拆包（FPartialNetBlob）、重发（FReliableNetBlobQueue）、包装（FRawDataNetBlob / FShrinkWrapNetBlob）等正交能力都能组合；
  • 用户加"自定义消息通道"只需实现 UNetBlobHandler，不必动引擎。

但再强调一次："一次性载荷"不包括属性稳态——那走自己的 State Stream 通路。完整全景（覆盖 7 种业务载荷、4 条主路）见下一节 §13.6。

自检 4 题
Q1. 客户端调 Server_Fire 时，参数 FireDirection 是怎么变成字节的？
→ UHT 生成的 Server_Fire_Validate/Implementation 调 CallRemoteFunction，里面走 FNetBitWriter+函数签名的 FRepLayout 把参数打包成 FNetBlob 的 payload。
Q2. 为什么 Reliable RPC 不能保证"低延迟"只能保证"最终到达且有序"？
→ 因为它会等前面序号先到，若卡住就得重发，RTT 本身是物理下限。要更快就只能用 Unreliable + 应用层幂等。
Q3. 目标 Actor 还没同步过来时 RPC 怎么办？
→ Writer 用特殊索引 AttachmentToObjectNotInScope 挂起，等 Filter 结果把它放进 Scope，再在 §09 后发出。
Q4. 一次 RPC payload 1.5 MB，UDP 单包 1200 B，Iris 会怎么处理？
→ 第一步在 FNetBlobManager::ProcessQueue 里就被 PreSerializeAndSplitNetBlob 拆开（Reliable 默认阈值 192 B，远小于 1.5 MB），产出一组 FPartialNetBlob（受 MaxPartCount × MaxPartBitCount ≈ 512 KB 上限约束，1.5 MB 这种极端体量实际会被 checkf 拦截或丢失部分）。第二步这组分片走 FReliableNetBlobQueue（reliable 时）跨帧发出，受 net.Iris.ReplicationWriterMaxAllowedPacketsIfNotHugeObject（默认 3）约束跨包数；若仍写不下，触发 bFallbackToHugeObjectPath，整批转 HugeObject 通道分多帧续写（net.Iris.ReplicationWriterMaxHugeObjectsInTransit 限同时在途数）。第三步客户端 FNetBlobAssembler 重组所有 part，整体 Deserialize 分派。注意：拆分动作发生在 RPC 入 Writer 队列之前，不依赖 HugeObject 触发；HugeObject 仅是写不下时的兜底通道。
Q5. 一个属性里含 FString PlayerName，每次同步真的会发整个字符串吗？
→ 不会。FString 字段的 NetSerializer 内部会调 FNetTokenStore::GetOrCreateToken，把 string 注册到独立的 NetToken 子流；主流里只发 4 字节左右的 token id。第二次起对端通过 token id 直接命中字典——这就是 13.6 图里 🟡 Token Stream 的作用。

写一个自定义 NetSerializer

已上线

Ch08 讲过 NetSerializer 的契约，Ch15 梳理过它是 L2 的核心。本章手把手写一个： FAngleAsInt8NetSerializer —— 把 float 角度 (0~360°) 压成 int8 (0~255 格)， 足以展示完整的 8 函数契约，也展示 Server 侧 Quantize/Serialize vs Client 侧 Deserialize/Dequantize 的对称性。

19.1 三个必备类型

SourceType
属性里的原始类型——这里就是 float。
Server 在 Quantize 阶段读它；Client 在 Dequantize 阶段写它。
QuantizedType
压缩态——这里是 uint8（1 字节，256 级）。StateBuffer 存的就是它；双端相同布局。
ConfigType（可选）
运行时参数，如"角度上限 / 是否 wrap"。若不需要就用 FNetSerializerConfig 空结构。

// AngleAsInt8NetSerializer.h —— 放在你的模块 Public 目录
#pragma once
#include "Iris/Serialization/NetSerializer.h"

namespace MyGame::Net
{
    struct FAngleAsInt8NetSerializerConfig : public UE::Net::FNetSerializerConfig
    {
        // 允许上层通过 TNetPropertySerializer 指定
        float MinDegrees = 0.f;
        float MaxDegrees = 360.f;
    };

    struct FAngleAsInt8NetSerializer
    {
        // 1) 三个必备 typedef
        typedef float                        SourceType;
        typedef uint8                        QuantizedType;
        typedef FAngleAsInt8NetSerializerConfig ConfigType;

        // 2) 契约：8 个核心成员函数（签名抄自 NetSerializer.h 里的 Args）
        static void Quantize (UE::Net::FNetSerializationContext&, const UE::Net::FNetQuantizeArgs&);
        static void Dequantize(UE::Net::FNetSerializationContext&, const UE::Net::FNetDequantizeArgs&);
        static void Serialize (UE::Net::FNetSerializationContext&, const UE::Net::FNetSerializeArgs&);
        static void Deserialize(UE::Net::FNetSerializationContext&, const UE::Net::FNetDeserializeArgs&);
        static bool IsEqual   (UE::Net::FNetSerializationContext&, const UE::Net::FNetIsEqualArgs&);
        static bool Validate  (UE::Net::FNetSerializationContext&, const UE::Net::FNetValidateArgs&);
        static void SerializeDelta(UE::Net::FNetSerializationContext&, const UE::Net::FNetSerializeDeltaArgs&);
        static void DeserializeDelta(UE::Net::FNetSerializationContext&, const UE::Net::FNetDeserializeDeltaArgs&);

        // 3) 必须在 cpp 里通过 UE_NET_DECLARE_SERIALIZER / UE_NET_IMPLEMENT_SERIALIZER 注册
    };

19.3 AngleAsInt8NetSerializer.cpp

Quantize · Server Dequantize · Client —— 一对对称变换：
#include "AngleAsInt8NetSerializer.h"
#include "Iris/Serialization/NetBitStreamReader.h"
#include "Iris/Serialization/NetBitStreamWriter.h"

namespace MyGame::Net
{
    static FAngleAsInt8NetSerializerConfig DefaultConfig;

    // ─── [Server] 从 float 压成 uint8 ───
    void FAngleAsInt8NetSerializer::Quantize(UE::Net::FNetSerializationContext&,
        const UE::Net::FNetQuantizeArgs& Args)
    {
        const auto& Cfg = *reinterpret_cast<const ConfigType*>(Args.NetSerializerConfig);
        const float Val = *reinterpret_cast<const SourceType*>(Args.Source);
        const float Span  = FMath::Max(Cfg.MaxDegrees - Cfg.MinDegrees, KINDA_SMALL_NUMBER);
        const float Norm  = FMath::Clamp((Val - Cfg.MinDegrees) / Span, 0.f, 1.f);
        *reinterpret_cast<QuantizedType*>(Args.Target) = static_cast<uint8>(Norm * 255.f + 0.5f);
    }

    // ─── [Client] 从 uint8 还原回 float ───
    void FAngleAsInt8NetSerializer::Dequantize(UE::Net::FNetSerializationContext&,
        const UE::Net::FNetDequantizeArgs& Args)
    {
        const auto& Cfg = *reinterpret_cast<const ConfigType*>(Args.NetSerializerConfig);
        const QuantizedType Q = *reinterpret_cast<const QuantizedType*>(Args.Source);
        const float Span  = Cfg.MaxDegrees - Cfg.MinDegrees;
        *reinterpret_cast<SourceType*>(Args.Target) = Cfg.MinDegrees + (Q / 255.f) * Span;
    }

    // ─── [Server] 把 uint8 写进 bits 流（固定 8 bits）───
    void FAngleAsInt8NetSerializer::Serialize(UE::Net::FNetSerializationContext& Ctx,
        const UE::Net::FNetSerializeArgs& Args)
    {
        const QuantizedType Q = *reinterpret_cast<const QuantizedType*>(Args.Source);
        Ctx.GetBitStreamWriter()->WriteBits(Q, 8);
    }

    // ─── [Client] 从 bits 流读出 uint8 ───
    void FAngleAsInt8NetSerializer::Deserialize(UE::Net::FNetSerializationContext& Ctx,
        const UE::Net::FNetDeserializeArgs& Args)
    {
        *reinterpret_cast<QuantizedType*>(Args.Target) =
            static_cast<QuantizedType>(Ctx.GetBitStreamReader()->ReadBits(8));
    }

    // ─── [Server] 脏判定：Ch10 Stage 9 Quantize 后调一次 ───
    bool FAngleAsInt8NetSerializer::IsEqual(UE::Net::FNetSerializationContext&,
        const UE::Net::FNetIsEqualArgs& Args)
    {
        const QuantizedType A = *reinterpret_cast<const QuantizedType*>(Args.Source0);
        const QuantizedType B = *reinterpret_cast<const QuantizedType*>(Args.Source1);
        return A == B;
    }

    // ─── [Server] 反射生成前做值域检查 ───
    bool FAngleAsInt8NetSerializer::Validate(UE::Net::FNetSerializationContext&,
        const UE::Net::FNetValidateArgs& Args)
    {
        const float V = *reinterpret_cast<const SourceType*>(Args.Source);
        return FMath::IsFinite(V);
    }

    // ─── Delta：简化版就是"不同则全量发"。真实场景可用差值压到 < 8 bits ───
    void FAngleAsInt8NetSerializer::SerializeDelta(UE::Net::FNetSerializationContext& Ctx,
        const UE::Net::FNetSerializeDeltaArgs& Args)
    {
        Serialize(Ctx, {Args.Version, Args.NetSerializerConfig, Args.Source});
    }
    void FAngleAsInt8NetSerializer::DeserializeDelta(UE::Net::FNetSerializationContext& Ctx,
        const UE::Net::FNetDeserializeDeltaArgs& Args)
    {
        Deserialize(Ctx, {Args.Version, Args.NetSerializerConfig, Args.Target});
    }

    // ★ 把以上打包成 FNetSerializer，注册到全局表
    UE_NET_IMPLEMENT_SERIALIZER(FAngleAsInt8NetSerializer);
}

// MyTeamFilter.cpp
void UMyTeamFilter::Filter(FNetObjectFilteringParams& P)
{
    // 每条连接对一批对象过滤一次。参数里给的是内部索引的 BitArrayView
    const UE::Net::FNetBitArrayView In  = P.OutAllowedObjects;  // 起手默认全过
    UE::Net::FNetBitArrayView Out = P.OutAllowedObjects;

    for (uint32 Idx : In)
    {
        const UObject* Obj = GetObjectForHandle(P.ObjectInfos[Idx].Handle);
        const AActor* A = Cast<AActor>(Obj);
        if (!A) continue;

        const int32 ObjTeam  = A->GetTeamId();
        const int32 ConnTeam = GetTeamForConnection(P.ConnectionId);

        if (ObjTeam != ConnTeam) {
            // 再看是不是隐身
            if (GetDefault<UMyTeamFilterConfig>()->bRespectStealth &&
                A->IsStealthedFrom(P.ConnectionId)) {
                Out.ClearBit(Idx);               // ← 核心：清位
            }
        }
    }
}

// MyImportancePrioritizer.h

    GENERATED_BODY()
protected:
    virtual void Prioritize(FNetObjectPrioritizationParams&) override;
};

// MyImportancePrioritizer.cpp
void UMyImportancePrioritizer::Prioritize(FNetObjectPrioritizationParams& P)
{
    const UE::Net::FReplicationView& View = P.View;              // Ch09 见过
    const FVector ViewPos = View.Views[0].Pos;

    for (uint32 Idx : P.ObjectIndices) {
        const FVector ObjPos    = GetObjectWorldLoc(Idx);
        const float   Imp       = GetObjectImportance(Idx);        // 项目自定义的优先级
        const float   DistSq    = FVector::DistSquared(ViewPos, ObjPos);
        const float   DistDecay = 1.f / (1.f + DistSq * 1e-4f);    // 距离平方反比

        P.Priorities[Idx] = Imp * DistDecay;
    }
}

G.1 FAQ · 常见问题

Iris 现在可以在生产里用吗？

Iris 仍属 Experimental（路径 Runtime/Experimental/Iris/ 即是证据）。官方 Fortnite BR / Fortnite Creative 等大规模项目已使用； 第三方项目需要评估 replay（Demo Net）、client-authoritative 预测、自定义 NetDriver 子类 三大兼容点——参见 Ch17 的 IrisNetDriverConfigs 白名单：可以做到"按 NetDriver 粒度逐步迁移"。

Iris 与 Mass / ECS 是什么关系？

Mass-Iris 不是替代关系而是复用：在 L7 换一个 UReplicationBridge 子类，把 Mass Entity 映射到 FNetRefHandle，上层的 L1~L6（序列化/过滤/调度）完全复用。详见 Ch16.6。

我自己的 USTRUCT 怎么参与复制？

三种方式，覆盖绝大多数场景：

  1. 让成员全部用内建 Serializer（最省事，Iris 自动合成 Struct Serializer）
  2. 写自定义 FNetSerializer，用 UE_NET_IMPLEMENT_SERIALIZER 注册——Ch19 有完整例子
  3. 做成 TArray<FMyItem> + FFastArrayReplicationFragment，自动支持增删 Delta

为什么要"量化 (Quantize)"？游戏内存不就是 POD 吗？

因为 UE 内存不全是 POD（FString / TArray / FSoftObjectPtr 都有元数据），且不适合按位 diff。量化后 QuantizedType 是紧凑 POD——memcmp 友好、逐位 Delta 可做、免反射批扫（见 Ch04 痛点⑤）。

ChangeMask 最大多少位？超大对象怎么办？

内联上限 LocalChangeMaskMaxBitCount = 64（见 ReplicationFragmentInternal.h）；超过则堆上分配。字段过多的 Actor 更优解是 拆 SubObject（每个独立 NetRefHandle）或 FastArray 让粒度细化。

没有 Iris 的项目切过去成本大吗？

按 Ch17 描述，最小改动：DefaultEngine.ini 里加 IrisNetDriverConfigs（逐 NetDriver 开启）+ 每个 Actor 加 RegisterReplicationFragments。
中等改动：自定义 RPC / 自定义 NetSerializer 迁移、ReplicateSubobjects 虚函数废弃改为 Bridge 注册。
大改动：依赖 UNetConnection::IsNetReady / UActorChannel 的底层游戏代码需要改走 UDataStream。

Iris 和 ReplicationGraph 能同时用吗？

不能。RepGraph 是 Legacy 时代的相关性优化层；Iris 的三级 Filter + Spatial Prioritizer 实现了 RepGraph 的所有关键能力（见 Ch16.7）。启用 Iris 后 RepGraph 配置被忽略。迁移方式：把 RepGraph 的 UReplicationGraphNode_GridSpatialization2D 对应到 UNetObjectGridFilter（Ch20）。

Iris 支持回放（Replay / Demo Net）吗？

目前建议：Demo NetDriver 保留 Generic，通过 IrisNetDriverConfigs 白名单只对 GameNetDriver 开 Iris。后续版本逐步迁移——这也是 Ch17.3 第③坑的答案。

Push Model 是 Iris 独有吗？

不是。MARK_PROPERTY_DIRTY_FROM_NAME 宏 Legacy 也支持，但只是"跳过扫描的性能优化"；Iris 把它升格为首等公民——Push-based 字段直接在 L5 DirtyNetObjectTracker BitArray 置位，没打脏就永远不会发。详见 Ch04 痛点④、Ch16.2。

我改了 Server，Client 的 OnRep_* 没回调，排查顺序？

严格按 Ch21.6 的 5 步流程图：
① Server stat Iris · DirtyObjects 是否涨 → 不涨则漏了 MARK_PROPERTY_DIRTY_FROM_NAME；
② net.Iris.Debug.FilteredObjectClassNames 看日志有没有命中；
③ Client stat Net · In.NbRPCs 是否涨；
④ 双端 NetTrace → Insights 对齐时间线；
⑤ C++ 断点 FReplicationWriter::WritePacket 与 FPropertyReplicationFragment::ApplyReplicatedState。

Iris 会多消耗内存吗？每个对象需要多少？

Iris 把"每连接 × 每对象"的复制状态压到量化态 POD + BitArray——大多数场景总内存反而下降（Ch04 痛点①的直接收益）。单对象内存 ≈ QuantizedTypeSize（所有字段之和）+ ChangeMaskBitCount/8 字节 + 每活跃连接一个 BitArray 位。典型 ACharacter 级对象 ~200 B 内。

属性同步和 RPC 都在 FNetBlob 里吗？Fragment 和 NetBlob 什么关系？

不都是。 FNetBlob 只装一次性载荷——RPC（FNetRPCBlob）/ 用户自定义 Attachment / 它们超阈值时的拆包碎片（FPartialNetBlob）/ 套膜（FShrinkWrapNetBlob）。属性稳态走的是独立的 State Stream：由 FReplicationProtocol::SerializeWithMask 直接驱动 NetSerializer 写主 BitStream，完全不经过 FNetBlob。唯一例外是 HugeObject 路径——单对象状态过大时整段 state 会被打包成 FRawDataNetBlob + 拆成 FPartialNetBlob 借道 Blob 通道。同一个对象包里，属性在前、Attachment 在后，并排共存。补充：FPartialNetBlob 不是 HugeObject 专属——常规 RPC 默认超过 192 字节就被 PreSerializeAndSplitNetBlob 拆，与"巨型对象"无关。

Fragment 和 NetBlob 不是上下游，而是两个正交的抽象：

  • FReplicationFragment（L4）= "某 UObject 里长期存在的可复制段的 schema + Poll/Apply 钩子"——管稳态。
  • FNetBlob（L3）= "一次性构造、发完即销毁的载荷"——管一次性消息。
  • 两者的唯一共同点是都用 FReplicationStateDescriptor + NetSerializer 作为底层序列化契约。

详细对照表见 §13.5。

自定义 Filter / Prioritizer 能放 Client 吗？

不行——L5 决策层只在 Server 启用（Ch15 模块 ⑦⑧⑨⑩ 全部标仅 Server）。"Client 想按距离隐藏某些 Actor"属于游戏层逻辑，请在 Client 的 Gameplay 代码里处理，而不是在复制层。
